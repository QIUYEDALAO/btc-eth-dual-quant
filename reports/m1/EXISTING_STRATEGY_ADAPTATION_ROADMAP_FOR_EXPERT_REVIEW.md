# 现成策略筛选、Freqtrade 兼容改造与验证路线报告书

- 日期：2026-07-20
- 状态：`proposal_for_expert_review_project_paused`
- 项目：`btc-eth-dual-quant`
- 当前执行权限：仅限专家审查本报告；不得下载候选、改造策略、回测、打开 OOS、连接 API 或交易

## 一、执行摘要

项目此前沿 U-04 至 U-24 连续设计新的横截面假设，并执行结果盲协议、数据资格与一次性 Paper 可行性观察。该流程保持了较强的防前视和证据完整性，但没有产生一个进入固定规则实现和正式回测的可用候选。继续创建 U-25、U-26 的预期边际价值已经很低。

建议停止“从零连续发明候选”的研究方式，改用“公开现成策略池 → 静态筛选 → 原样兼容移植 → 统一 IS 回测 → 有限改造 → 独立验证”的工程路线。这样可以复用已经存在的经济逻辑与 Freqtrade 代码，将主要资源投入兼容性、风险控制和可信验证。

本报告只提出新路线，不授权执行。项目保持暂停；OOS、dry-run、API 和 live 必须继续单独授权。

## 二、当前项目事实与问题

### 2.1 已完成事实

- V4 公共数据、点时成员、lifecycle 与 invalid-interval authority 已形成冻结证据链。
- Freqtrade 已被接受为单腿现货研究、回测和未来自动化框架。
- U-24 唯一 Paper 观察以 `failed_feasibility` 关闭；OOS 未打开，未产生正式策略收益。
- 当前工作区包含尚未统一提交的 U-25 授权草稿；项目已经由用户暂停。

### 2.2 原路线的主要问题

- 每个候选都重复生成设计、协议、审查、资格、观察和治理材料，开发成本高。
- U-04 至 U-24 没有候选进入可用 Freqtrade 策略阶段，实际产品进展不足。
- 过度强调逐候选治理，挤占了策略代码审查、兼容移植和统一回测的时间。
- 连续创造新假设并不保证比审查成熟公开策略更容易获得可执行结果。

## 三、建议目标

在不降低因果、数据和风险标准的前提下，从公开策略中筛选一批可解释候选，原样移植到固定 Freqtrade 版本，使用同一数据、交易对、成本和指标进行公平比较；只允许少量预注册改造，最终冻结一个候选进入唯一 OOS 决策。

本阶段目标不是直接实盘，而是得到：

1. 一个有明确来源与许可证的策略候选池；
2. 一组可以被固定 Freqtrade 版本准确加载的原样兼容实现；
3. 可复现、无明显前视偏差的统一 IS 对比结果；
4. 至多一个通过冻结 Gate 的最终候选；
5. 在 OOS、dry-run 和 live 权限边界前强制暂停。

## 四、首批公开来源

建议优先审查以下公开仓库，但不因知名度直接认定策略有效：

- [freqtrade/freqtrade-strategies](https://github.com/freqtrade/freqtrade-strategies)：官方组织的基础策略合集，适合作为简单基准。
- [TheoBrigitte/freqtrade](https://github.com/TheoBrigitte/freqtrade)：包含策略、配置、回测结果和 lookahead 检查流程。
- [iterativv/NostalgiaForInfinity](https://github.com/iterativv/NostalgiaForInfinity)：活跃的大型社区策略，但复杂度、DCA 和极端回撤风险需要重点审查。
- [nateemma/strategies](https://github.com/nateemma/strategies)：策略与测试工具丰富，但框架依赖和超参数复杂度较高。

每个引入候选必须记录仓库 URL、精确 commit、文件 SHA-256、许可证、原始配置、依赖、交易模式、timeframe 和作者声明。不得只复制无法追溯来源的单个文件。

## 六、静态筛选规则

### 6.1 优先保留

- Binance spot、USDT、long/cash 或可安全降级为 long/cash；
- 经济逻辑可以用几段文字准确解释；
- 指标与数据需求可以从冻结 OHLCV 或明确公共数据获得；
- 有明确止损、退出和仓位边界；
- 代码规模和依赖可审计；
- 能在固定 Freqtrade 版本下稳定加载。

### 6.2 默认排除

- 强制 futures、short、leverage 或跨交易所多腿；
- 无限或高层数 DCA、亏损加仓、无总风险上限；
- 使用未来数据、同一根收盘信号按该收盘成交、负 shift 或全样本统计；
- 必须调用私有 API、付费信号、远程模型或未知二进制；
- 只提供截图，没有源码、参数、commit 或许可证；
- 通过大量 hyperopt 参数才能定义策略本身；
- 许可证禁止项目预期使用方式。

NostalgiaForInfinity 等复杂策略可以保留为审查对象，但不得在未拆清 DCA、仓位和退出风险前进入统一回测。

## 七、兼容改造边界

“原样兼容移植”允许：

- 升级旧版 Freqtrade strategy interface；
- 将 buy/sell 列迁移为 enter/exit；
- 修复导入路径、指标 API、informative pair 合并与 timeframe 声明；
- 将 Pine Script 或普通 pandas 逻辑等价改写为 Freqtrade Python；
- 将信号执行修正为严格下一合格开盘；
- 增加只用于安全 fail-closed 的空数据和 lifecycle 检查。

原样阶段禁止：

- 根据本项目回测改变阈值、周期、ROI、止损或仓位；
- 删除亏损信号、增加事后过滤器或挑选赢家交易对；
- 以 Hyperopt 结果替代原策略；
- 把兼容性修复伪装成经济规则不变。

任何影响交易集合或风险收益的变化必须进入阶段 8 的独立有限改造合同。

## 八、统一验证建议

最终数值 Gate 应由专家在首次结果前确认。建议至少冻结：

- 固定 Freqtrade 版本、spot 模式、初始资金和最大并发仓位；
- 固定点时 active-universe pairlist，不使用当前成员回填；
- IS/OOS 时间边界保持密封；
- Base `0.30%` 与 Cost×2 `0.60%` roundtrip 成本，并增加保守滑点；
- Base 与 Cost×2 总收益、daily-MTM Sharpe、PSR、MaxDD、交易数、年度/季度稳定性；
- 删除最佳三笔后的收益；
- 与相同风险预算的 active-universe benchmark 对比；
- `lookahead-analysis`、`recursive-analysis`、交易导出和独立重定价；
- 参数数量、单币种贡献、单年份贡献和最大持仓集中度。

可沿用历史参考 Gate：Base/Cost×2 均盈利、交易数至少 56、daily-MTM Sharpe 至少 1.0、PSR 至少 0.95、MaxDD 不超过 15%、删除最佳三笔后不亏、四个子区间至少三个盈利，并胜过同风险 benchmark。专家应确认这些数值是否适合“多公开策略统一筛选”，以及是否需要增加多重比较惩罚。

## 九、防止换一种方式过拟合

公开策略筛选本身也会增加试验次数，因此必须：

- 在首次统一回测前冻结候选清单和淘汰 Gate；
- 记录所有进入回测的候选，不能只登记赢家；
- 将原样比较与有限改造视为不同 trial；
- 不允许根据失败结果不断更换 timeframe、pairlist 或成本；
- OOS 只允许最终一个候选打开一次；
- 最终统计考虑候选数量和历史 opened-OOS trial count。

## 十、Sol 与 Codex 协作流程

1. Sol 制定每个完整阶段的计划、边界和验收 Gate。
2. Codex 在阶段内连续实现、修 Bug、运行测试，不因普通可解决问题反复暂停。
3. Codex 完成本地结果后交由 Sol 做阶段 Review 和验收。
4. Codex 按 Sol Review 集中修改并重新执行统一验证。
5. Sol 最终确认整个交付范围。
6. 最后统一执行一次 `commit → push → PR`；确认前不得提前发布。

## 十二、建议决策

建议专家批准阶段 0～5：旧主线收口、公开候选清单、静态筛选、5～8 个 exact-source 冻结、原样 Freqtrade 兼容移植以及因果/框架检查。阶段 6 首次读取统一 IS 性能结果，应在专家同时确认候选数量、成本、benchmark、数值 Gate 和多重比较处理后再执行。
